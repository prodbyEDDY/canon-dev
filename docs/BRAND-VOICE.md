# Brand voice — голос документации canon-dev

Этот документ — про то, как звучит документация, генерируемая через `canon-dev`. Тон важен: канон — это язык, на котором архитектор говорит с разработчиком через время. Если язык расхлябан, сам канон становится расхлябанным.

---

## Тон

**Спокойный, точный, структурный.** Канон — это не маркетинг и не учебник. Это рабочий документ, который читают раз в месяц чтобы найти конкретный инвариант или принять решение по edge case.

Не должно быть:
- Восклицательных знаков
- Эмодзи в тексте (заголовки и навигационные пункты — допустимо как графический разделитель, но не как пунктуация)
- Суперлативов («лучший», «революционный», «самый эффективный»)
- Маркетинг-клише («просто как 1-2-3», «по щелчку пальцев»)
- Стартап-сленга («давайте захолим эту фичу», «MVP-кратко»)
- Бесцельных мета-комментариев («мы тут хотели сказать что…»)

Должно быть:
- Прямые декларативные утверждения («Модуль X отвечает за Y»)
- Императивные правила («Поле Z обязательно для статуса W»)
- Em-dash для связки идей («Канон выше кода — потому что код это интерпретация одной головы»)
- Списки вместо прозы там где уместно

## Структура канона

Любой канон следует общей структуре. Не все секции обязательны для каждого модуля, но порядок одинаков:

```
1. Назначение и место в системе
2. Что модуль делает / Чего модуль не делает
3. Пользователи и типовые задачи
4. Главные архитектурные правила
5. Главные сущности и их разведение
6. Lifecycle статусов
7. Источники истины и направление потока данных
8. Связь с другими модулями
9. Антидубли и блокировки
10. Типовые сквозные сценарии
11. Что в зоне out-of-scope
12. Тест-кейсы / приёмочные критерии
```

Не все 12 секций нужны — для маленького модуля достаточно 1-7. Но порядок не меняется.

---

## Структура ТЗ

ТЗ детальнее канона. Включает:

```
1. Контекст и scope (повтор канон-выжимки)
2. Структура страницы / экрана (для UI-модулей)
3. Поля схемы данных
4. API-контракты (по endpoint'у)
5. Бизнес-формулы (если применимо)
6. Lifecycle и переходы статусов
7. Обязательные поля по статусу
8. Антидубли (на уровне схемы и сервиса)
9. Аудит-логирование
10. Edge cases и их обработка
11. Out-of-scope явно
12. Связи с другими модулями (контракты)
```

## Голос промпта для агента-реализатора

Промпт — это другой жанр, чем канон. Канон описывает «как должно быть в системе». Промпт описывает «что конкретно нужно сделать в этом PR».

Тон промпта:
- **Императивный.** «Создай таблицу X. Добавь endpoint Y. Замени Z на W.»
- **Дисциплинированный.** В каждой секции — что входит, что НЕ входит. Out-of-scope явно.
- **С референсами.** Каждое правило — цитата из канона или ссылка на §раздела.
- **Самодостаточный.** Агент должен выполнить промпт без обращения к чату или другим документам.

Структура промпта:
```
0. Контекст и scope
1. Жёсткие правила (брендовые, кодовые)
2. Schema additions
3. Бизнес-логика / расчёты
4. API-контракты
5. UI-изменения
6. Cross-module integration
7. Тестовые сценарии (самопроверка агента)
8. Pre-merge чеклист
9. Out-of-scope
10. PR-описание шаблон
11. Финальная заметка / акценты
```

## Примеры

### ❌ Плохо (маркетинг-голос)

> Налоги — это супер-важный модуль для управления налогами вашей компании! С его помощью вы можете легко и быстро начислять, оплачивать и отслеживать все налоговые обязательства. Это просто 🎯

### ✅ Хорошо (канон-голос)

> Модуль «Налоги» — рабочий обязательственный слой раздела «Финансы». Он начинается от налогового профиля компании, ведёт налоговые обязательства и платежи по периодам, передаёт финансовые последствия только через единый реестр финансовых операций. Не дублирует бухгалтерскую систему, не подменяет платёжный модуль.

### ❌ Плохо (расплывчатый промпт)

> Сделай модуль налогов как обсуждали. Используй стандартные паттерны. Если что-то непонятно — посмотри как сделано в других местах.

### ✅ Хорошо (самодостаточный промпт)

> Создай таблицу `tax_obligation` со следующими полями: [...].
> CHECK constraints: amount_kopecks ≥ 0, paid_kopecks ≤ amount_kopecks.
> Endpoint POST /api/tax/obligations с Zod-схемой [...].
> При status='approved' — автоматическое создание calendar event (см. canon §24.1).
> Out-of-scope в этом PR: payments lifecycle, проведение в financial_operation. Это в следующем PR фазы.

## Правила копирайта

1. **Em-dash с пробелами** — основной соединитель идей.
2. **«Ёлочки»** для названий продукта/модуля в running copy: «Налоги», «Финансы».
3. **Sentence case** для заголовков: «Что получает бизнес», не «Что Получает Бизнес».
4. **Числа:** тонкий пробел между разрядами (`4 750 000 ₽`), запятая десятичная (`21,4%`), `₽` через non-breaking space.
5. **Никаких заглавных в середине предложения** кроме имён собственных.
6. **Аббревиатуры** в первый раз с расшифровкой, потом просто аббревиатура: «налог на доходы физических лиц (НДФЛ)», далее — НДФЛ.
7. **Кавычки** для цитат — «ёлочки» русские, не английские "квадратные".

## Цена голоса

Если документация не следует этому голосу — она остаётся, но теряет силу. Команда читает её один раз, помечает «надо переписать», и больше не возвращается. Канон должен звучать как строительный код, а не как промо-материал. Тогда им пользуются.
