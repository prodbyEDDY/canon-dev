# Банк уточняющих вопросов по доменам

> Готовые вопросы для Phase 2 скилла `braindump-to-canon`. Группированы по доменам. Не задавай все — выбирай релевантные для проекта.

---

## Multi-tenancy и доступ

1. Один пользователь — одна организация, или может принадлежать нескольким?
2. Если несколько — как переключается между ними? Через UI? Через subdomain?
3. Кто создаёт организацию? Любой пользователь? Только админ?
4. Сколько пользователей в одной организации (типично, максимум)?
5. Как пригласить нового пользователя — email-link? Pre-created account?
6. Может ли пользователь видеть данные **другой** организации? Никогда? Только при явном permission?
7. Есть ли «глобальный супер-админ» которому видны все организации?

## Деньги (если применимо)

1. Какая валюта основная? Мульти-валюта поддерживается?
2. Где деньги — одна сущность? Несколько (счета, кошельки)?
3. Двойная запись (бухгалтерская) или single-entry?
4. Округление: до копейки? До рубля? Banker's rounding или half-up?
5. Кто меняет суммы? Только пользователь? Системные расчёты?
6. Есть ли «закрытие периода» когда суммы становятся неизменяемыми?
7. Как храним суммы — integer (копейки) или decimal? **Никогда float.**
8. Конвертация валют — ручная или auto?
9. Журнал изменений сумм — обязателен?

## Lifecycle сущностей

1. У каких сущностей есть статусы (states)?
2. Перечисли статусы для каждой и переходы между ними.
3. Какие переходы запрещены? (например, нельзя сразу из draft в archived без active)
4. Есть ли «закрытие» (frozen state)?
5. Как переоткрыть закрытую сущность? Кто может?
6. Есть ли soft-delete или физическое удаление?
7. Если soft-delete — как helps вернуть случайно удалённое?

## Аудит и безопасность

1. Какие действия должны логироваться обязательно?
2. Что хранится в журнале — только событие или old/new значения?
3. Как долго хранится журнал? Удаляется когда-нибудь?
4. Кто видит журнал? (роль)
5. Есть ли «критические действия» требующие подтверждения второго пользователя?
6. 2FA / MFA — обязательно для всех? Только админов? Опционально?
7. Сессии — сколько живут? Можно ли убить чужую сессию админу?
8. Как обрабатываем подозрительные действия (множественные failed logins, странная geo-локация)?

## Внешние интеграции

1. С какими внешними системами интегрируемся?
2. Это блокирующая зависимость или nice-to-have?
3. Что делаем когда внешняя система недоступна — failover, retry, локальный mode?
4. Webhook'и от внешних систем — есть ли? Как обрабатываем дубли?
5. API-ключи внешних — где храним (env vs secret manager)?
6. Rate-limits внешних — учитываем?
7. Audit-log включает ли наши вызовы внешних API?

## Регуляторика

1. Какому законодательству подчиняемся (152-ФЗ, GDPR, HIPAA, SOC2)?
2. Где хранятся персональные данные пользователей? Гео-ограничения?
3. Право на забвение — реализовано? Как?
4. Согласие на обработку — где собираем, как храним?
5. Возрастные ограничения (COPPA если applicable)?
6. Industry-specific (финансовая лицензия, медицинская)?

## Документы и файлы

1. Какие файлы загружают пользователи?
2. Лимиты размеров?
3. Допустимые форматы?
4. Как храним — local fs, S3, blob storage?
5. Версионность файлов — нужна?
6. Кто видит чьи файлы? Permissions?
7. Срок хранения?
8. Шифрование at-rest?

## Производительность и масштаб

1. Сколько пользователей одновременно ожидаем (concurrent)?
2. Сколько запросов в секунду — в пиковые часы?
3. Сколько записей в крупнейших таблицах ожидается через год?
4. Какие операции могут быть тяжёлыми (отчёты, агрегация)?
5. Нужен ли кэш? На каком уровне (CDN, app, DB)?
6. Real-time updates — где? (WebSocket / SSE / polling)
7. Локализация / гео-распределение — есть?

## ИИ / LLM (если применимо)

1. Где в продукте LLM? (chat, classification, generation)
2. Какие данные пойдут в провайдера? Persondata?
3. Кэш промптов — нужен?
4. Streaming — да/нет?
5. Что делаем если модель отвечает плохо / слишком долго?
6. Hallucination-risk — где особенно опасен?
7. Промпт-инъекция — продумали?

## Уведомления

1. Какие события рождают уведомления?
2. Каналы — email, push, in-app, SMS?
3. Пользователь может отключить категории? Все?
4. Цифровая подпись email-ов?
5. Локализация уведомлений?
6. Транзакционные vs marketing — разделены?

## Тесты и качество

1. Уровни тестирования: unit / integration / E2E — какие нужны?
2. Что обязательно покрыто тестами (критичные расчёты, безопасность)?
3. Smoke-тесты на проде — есть?
4. Manual QA на каждый релиз — есть?
5. Beta-программа / canary releases — нужны?

## Backup и DR

1. Backup БД — частота? Retention?
2. Тестируем ли восстановление — регулярно?
3. RTO (recovery time objective) — какая цель?
4. RPO (recovery point objective) — допустимая потеря данных?
5. Если потеряли весь регион — что делаем?

## Onboarding и UX

1. Новый пользователь — какой первый экран?
2. Первый шаг что должен сделать?
3. Pre-filled demo-данные — да/нет?
4. Гид / tour — нужен?
5. Помощь / docs — встроена в продукт или отдельный сайт?
6. Чат с поддержкой — есть? Через что?

## Pricing и billing (если SaaS)

1. Тарифы — какие?
2. Триал — есть? Длительность?
3. Платёжная система?
4. Автоматическое списание?
5. Что происходит при просрочке оплаты — read-only mode? Сколько дней grace?
6. Налоги (НДС, sales tax) — учитываем? Где считаем?
7. Receipts / invoices — генерируем?
8. Возвраты — поддерживаем?

## Мобильное

1. Mobile web (responsive) — обязательно?
2. Native app — планируется?
3. Offline-mode — нужен?
4. Push-нотификации — нужны?
5. Камера / геолокация — используем?

## Как использовать банк

В Phase 2 скилла `braindump-to-canon`:

1. Прочти первичный braindump
2. Определи 3-5 наиболее релевантных доменов из этого банка
3. Возьми по 5-7 вопросов из каждого, адаптируй под продукт
4. Задавай раундами (5-7 вопросов за раз, не 30)
5. Фиксируй ответы прямо в драфт канона
6. Если ответ «не знаю» — предложи 2-3 варианта с trade-off, или фиксируй как **открытый вопрос**

---

## Запрещённые техники

❌ **Bulk-questionnaire** — «вот вам 50 вопросов, заполните за неделю». Никто не заполнит.

❌ **Лидирующие вопросы** — «вы ведь хотите multi-tenancy через row-level security, правда?». Спроси нейтрально: «Как разделяем данные между организациями?»

❌ **Технические термины без объяснения** — пользователь может не знать что такое RPO. Объясни в скобках или замени термином.

❌ **Спрашивать ради проформы** — если ответ очевиден из контекста, не задавай. Двигайся быстро.

✅ **Признавать «не знаю»** — пользователь может реально не знать (это первая попытка продукта). Тогда твоя работа — предложить варианты с trade-off, не настаивать на ответе сейчас.
