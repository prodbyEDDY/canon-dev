# Findings classification

> Гайд по классификации severity для findings в `canon-review`. Severity влияет на приоритет фикса и блокирующий статус.

---

## 🔴 Critical

Finding получает **Critical** если соответствует **хотя бы одному** из критериев:

1. **Производственный риск:** без фикса в production произойдёт катастрофа — потеря данных, утечка между tenants, неверные денежные расчёты, нарушение compliance.

2. **Архитектурно-неправильно:** дальнейшая разработка опирается на этот gap, и через несколько фаз потребуется большой рефакторинг с миграцией данных.

3. **Безопасность:** уязвимость, leak PII, unauthorized access path.

4. **Денежная корректность:** float для денег, отсутствие journal append-only, hardcoded налоговые ставки.

5. **Multi-tenant нарушение:** возможен cross-tenant leak.

### Примеры

> ❌ В каноне Финансы написано «суммы хранятся как decimal(10,2)» — это финансовая float-precision. Должно быть integer-копейки.

> ❌ Канон Аналитика говорит «дашборд читает из БД» — но не указывает источник истины. Это означает что разработчик может выбрать «удобный» источник, что приведёт к рассинхрону.

> ❌ Multi-tenancy: канон Контрагенты не упоминает organization_id. Любой пользователь сможет видеть контрагентов всех организаций.

### Действие
**Блокирует разработку.** Перед началом phase 1 — обязательно закрыть.

---

## 🟡 Major

Finding получает **Major** если:

1. **Серьёзный недосмотр:** не катастрофично, но через 3-6 месяцев потребует рефакторинга или дизайн-смены.

2. **Lifecycle gap:** не описан важный переход или edge case.

3. **Performance / scale:** ожидаемый объём данных не учтён.

4. **Cross-module integration не описан:** разработчик будет домысливать.

5. **Antidubli отсутствует** (но не для денежных операций — те Critical).

### Примеры

> ⚠️ Канон Документы описывает создание/изменение, но не описывает архивирование. Через несколько месяцев потребуется добавить archive-flow с миграцией данных.

> ⚠️ Канон Задачи описывает статусы (active/done), но не описывает что происходит если ответственный сотрудник удалён. Edge case без обработки.

> ⚠️ Pagination не описан для list-endpoint Реестра договоров. При 10k+ записях query упадёт.

### Действие
**Желательно закрыть до конца phase 1.** Можно начать phase 1 с Major findings, но они должны быть в backlog.

---

## 🟢 Minor

Finding получает **Minor** если:

1. **Documentation quality:** структура неочевидная, terminology inconsistent.

2. **Missing example:** в каноне говорится про сложную формулу без примера.

3. **Cross-link broken:** ссылка на другой канон не работает.

4. **Typo / поломанный markdown.**

5. **Brand-voice violation** (одиночные эмодзи, восклицание, маркетинг-фраза).

### Примеры

> 🟢 В каноне Финансы термин «контрагент» используется как «партнёр» в одном месте. Разнобой.

> 🟢 Раздел «Расчёт НДС» описывает формулу, но не приводит ни одного учебного примера.

> 🟢 Cross-link `[см. канон X]` ведёт на удалённый файл.

### Действие
**Закрывать постепенно** при следующем touch'е соответствующего канона. Не блокирует phase 1.

---

## 💡 Suggestion

Finding получает **Suggestion** если:

1. **Improvement, not gap:** канон уже описывает функциональность, но можно улучшить ясность.

2. **Optional feature:** функция не обязательна, но улучшит DX/UX.

3. **Refactor opportunity:** можно объединить дублирующиеся части или вынести в общий слой.

### Примеры

> 💡 Раздел про lifecycle можно вынести в отдельный sub-канон (`Канон_Lifecycle.md`) и ссылаться из канонов модулей. Сейчас lifecycle описан в каждом каноне с похожими формулами.

> 💡 Можно добавить раздел «Common patterns» в главный канон с примерами реализации (как `useFinanceRefresh` pattern из реального проекта).

### Действие
**Делается когда есть time/budget.** Не приоритет.

## Decision tree для классификации

```
Это конкретный пробел или абстрактное «было бы лучше»?
├─ Конкретный пробел
│   │
│   Влияет на корректность системы (деньги, security, multi-tenant)?
│   ├─ Да → 🔴 Critical
│   └─ Нет
│       │
│       Lifecycle / integration / scale gap?
│       ├─ Да → 🟡 Major
│       └─ Нет → 🟢 Minor
│
└─ Абстрактное улучшение → 💡 Suggestion
```

---

## Anti-patterns в классификации

❌ **«Всё Critical»** — теряется приоритезация. Если 30 Critical findings — пользователь не знает с чего начинать.

❌ **«Всё Suggestion»** — занижение severity, реальные риски пропадают в шуме.

❌ **Subjective severity** — «мне не нравится формулировка» = это не finding в canon-review. Это совет автору. Если язык реально мешает пониманию — Minor. Если нет — не репорти.

❌ **Severity по personal preference** — например, «я бы выбрал другую DB». Это не canon-review область, это stack decision (другой скилл).

✅ **Severity на основе критериев** — пройди по списку выше, поставь severity по конкретному критерию.

## Калибровочные примеры

### Не Critical, а Major

> «В каноне Финансы lifecycle оплат описан только для happy path, не описано что при отмене оплаты, и что при возврате денег».

— Это lifecycle gap. Major. Не Critical потому что: (а) можно начать разработку с happy path, (б) баг при отмене найдётся в QA, (в) фикс потребует доработки канона + кода, но не migration.

### Не Major, а Minor

> «Раздел про антидубли использует термин "идемпотентность" без объяснения».

— Documentation quality. Minor. Не Major потому что: (а) разработчик с опытом поймёт термин, (б) можно добавить определение в глоссарий, (в) функционально ничего не сломается.

### Critical, не Major

> «Канон Налоги описывает что НДФЛ платится из ФОТ, но не описывает антидубль с tax-payment».

— Антидубль для денежной операции = Critical. Без него в production будет двойной расход в ОПиУ. Найдётся через бухгалтера который заметит расхождение в отчёте на 5 млн.

## Как использовать

В Phase 4 скилла `canon-review`:

1. Каждый finding из Phase 2-3 классифицируй по этому гайду
2. Если сомневаешься — задай один из критериев напрямую: «производственный риск»? «архитектурный долг»? «documentation gap»?
3. Записывай в отчёт строго по severity
4. В summary укажи total counts: `Critical: X, Major: Y, Minor: Z, Suggestions: W`

Это даёт пользователю чёткое понимание объёма работы и приоритета.
