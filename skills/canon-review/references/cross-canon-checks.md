# Cross-canon consistency checks

> Конкретные проверки на консистентность **между** канонами. Используется в Phase 3 скилла `canon-review`.

Если каждый канон проверяется отдельно (Phase 2), то Phase 3 — это проверка их вместе как **системы**.

---

## C1. Терминология (Glossary consistency)

### Что проверять

- Одна сущность называется одинаково в всех канонах?
- Если есть алиасы — задокументированы?
- Английские термины переводятся одинаково (например, `counterparty` всегда «контрагент», не иногда «партнёр»)?

### Метод

1. Создай ad-hoc глоссарий: для каждого ключевого термина — где встречается
2. Найди расхождения

### Пример finding

> X-01. Терминология «контрагент» vs «партнёр»
> - В Канон_Финансы используется «контрагент»
> - В Канон_Документы используется «партнёр»
> - В Канон_Контрагенты используется «контрагент»
> - **Рекомендация:** унифицировать на «контрагент». Glossary: `docs/КАНОНЫ/GLOSSARY.md`.

## C2. Lifecycle alignment

### Что проверять

Если модуль A создаёт сущность которую модуль B изменяет:
- Статусы согласованы?
- Что A считает «final» — то же что и B?
- Транзакционность работает (оба меняются вместе или ни один)?

### Пример finding

> X-02. Расхождение lifecycle на стыке Финансы ↔ Налоги
> - Канон_Финансы: operation.status может быть {draft, posted, cancelled}
> - Канон_Налоги: tax_payment может перевести operation в status='paid' — но такого статуса нет в Финансах
> - **Симптом:** компиляция или runtime error в production
> - **Рекомендация:** уточнить — либо Налоги используют существующие статусы Финансов, либо Финансы расширяют enum.

---

## C3. Источники истины — без дубликатов

### Что проверять

Один тип данных не может быть «источником истины» в двух местах. Проходишь по каждому типу данных и проверяешь:

- Где он создаётся?
- Где он мутируется?
- Где он только читается?

### Пример finding

> X-03. Двойной источник истины для balance счетов
> - Канон_Финансы: «Источник истины для остатка по счёту — financialOperation (агрегат)»
> - Канон_Платежи: «Поле financeAccount.balance — текущий остаток»
> - **Конфликт:** balance вычислимый из financialOperation vs хранимый
> - **Рекомендация:** определить — кеш или primary. Если кеш — описать invalidation. Если primary — financialOperation становится derived.

## C4. Антидубль на стыках

### Что проверять

Если модуль A может передать данные в модуль B:
- В обоих канонах описан антидубль одинаково?
- Кто отвечает за идемпотентность ключа: A (генерирует) или B (проверяет)?

### Пример finding

> X-04. Антидубль на стыке ФОТ ↔ Налоги
> - Канон_ФОТ: «При утверждении payroll-cycle создаются НДФЛ-обязательства в Налогах»
> - Канон_Налоги: НЕ упоминает антидубль для obligations с source='payroll'
> - **Симптом:** при retry approve будут дубли НДФЛ-обязательств
> - **Рекомендация:** добавить в Канон_Налоги раздел «Антидубль для payroll-source», использовать unique partial index `(taxPeriodId, sourceEntityType, sourceEntityId)`.

## C5. Стек consistency

### Что проверять

Все каноны опираются на тот же стек (`STACK.md`)?

- Если канон упоминает технологию которой нет в стеке — конфликт
- Если стек обновлён — все каноны в курсе?

### Пример finding

> X-05. Канон_Аналитика упоминает MongoDB для time-series
> - В STACK.md primary DB — PostgreSQL 18
> - В Канон_Аналитика §7: «time-series данные хранятся в MongoDB»
> - **Конфликт:** MongoDB не в стеке
> - **Рекомендация:** либо обновить стек (явное решение), либо переписать §7 канона на PG (TimescaleDB extension или partitioning).

---

## C6. Cross-module правила (RBAC, audit, naming)

### Что проверять

Сквозные правила должны применяться ко всем модулям одинаково:

- Audit-logging в каждом модуле использует ту же структуру?
- Authorization patterns — одинаковые?
- API-pattern (session check / orgId / Zod / log) — соблюдается всеми?
- Naming conventions (snake_case в DB, camelCase в TS) — последовательно?

### Пример finding

> X-06. Audit-logging inconsistency
> - Канон_Финансы: использует `system_audit_log`
> - Канон_Налоги: создаёт собственные audit-log таблицы (`tax_profile_audit_log`, `tax_period_audit_log`)
> - **Это валидно или нет?** Зависит от продуктового решения. Если оба используются — должно быть **явно описано** в главном каноне (например: «system_audit_log для cross-cutting events, module-specific audit-log для domain-events»). Если не описано — конфликт.
> - **Рекомендация:** явно описать стратегию в главном каноне §4.

---

## C7. Versioning справочников

### Что проверять

Если несколько канонов обращаются к нормативным справочникам (ставки, тарифы, лимиты) — стратегия версионности одинакова?

- Структура `validFrom`/`validTo` стандартизирована?
- Reading версии «актуальной на период» — одинаково реализовано?

---

## C8. Open questions blocking

### Что проверять

Открытые вопросы из `OPEN-QUESTIONS.md`:
- Какие блокируют какие модули?
- Если Q1 говорит «решим позже» — но без него Канон_X.md не может закрыться, это критическая ситуация

### Пример finding

> X-07. Q-03 блокирует phase 1 разработку
> - Q-03: «Multi-currency support — поддерживаем или нет?»
> - Помечена как «открытая»
> - Но Канон_Финансы требует ответа на этот вопрос для CHECK constraint на `currency` field
> - **Решение:** нужно закрыть Q-03 до начала phase 1, либо явно сказать «v1 = single-currency, multi в v2».

---

## C9. Dependencies в карте модулей

### Что проверять

В главном каноне есть карта зависимостей. Реальная карта из канонов модулей соответствует?

- Канон_X говорит что зависит от Канон_Y — Y действительно existsет?
- Дуговые dependencies (A ↔ B mutual) — допустимы или нужно разделять?

### Пример finding

> X-08. Циклическая зависимость Аналитика ↔ Финансы
> - Канон_Финансы §16: «провёденные операции попадают в финансовую аналитику»
> - Канон_Аналитика §3: «аналитика может писать признаки в первичные модули» (предполагаемое)
> - **Циклическая запись.** Источник истины становится неопределённым.
> - **Рекомендация:** Аналитика — read-only по канону §3 главного. Уточнить.

## Метод Phase 3 в двух шагах

### Шаг 1. Construct cross-canon matrix

Для каждой пары канонов составь короткую таблицу:

| Канон A | Канон B | Что A отдаёт B | Что B берёт у A | Конфликты? |
|---|---|---|---|---|
| Финансы | Налоги | financial_operation, finance_account | references | none |
| Финансы | Аналитика | aggregated metrics | read-only | none |
| ФОТ | Налоги | НДФЛ/insurance суммы из payrollRecords | tax_obligations с source='payroll' | check antidubli |
| Контрагенты | Финансы | counterparty references | optional FK | none |

### Шаг 2. Pass through C1-C10

Для каждой пары — пройди по check'ам выше.

---

## Финальная мысль

Cross-canon checks — самая часто пропускаемая часть `canon-review`. Каждый канон смотрят отдельно, и кажется что всё OK. А когда модули встречаются в коде — выясняется что Канон_A считает X источником, а Канон_B считает X производным.

Не пропускай Phase 3. Это занимает 1-2 часа дополнительно, но предотвращает рефакторинг через 6 месяцев.
