# Workflow — как скиллы соединяются

Этот документ показывает **полный pipeline** от первичной идеи до production-кода с указанием артефактов на каждой стадии.

---

## Стадии pipeline

### Стадия 0 — Pre-existing materials

То что у пользователя есть на старте:
- Свободный braindump (текст / голосовая запись / список идей)
- Возможно: бренд-материалы (логотип, гайдлайн, презентация)
- Возможно: техническое предпочтение по стеку
- Возможно: фрагменты существующей документации

**Артефакт:** свободные материалы в любом формате.

### Стадия 1 — Braindump → Canons (`braindump-to-canon`)

Структурированное интервью, превращающее свободный материал в каноническую документацию:

- **Канон верхнего уровня** — назначение продукта, аудитория, главные инварианты
- **Каноны модулей** — для каждого модуля: назначение, главные сущности, lifecycle, антидубли, связь с другими модулями
- **ТЗ модулей** — детальная техническая спецификация (поля схем, API-контракты, UI-структура)
- **Stack-decision доклад** — выбор технологического стека с обоснованием

**Артефакт:**
```
docs/КАНОНЫ/
├── 00-Главный-канон.md
├── Канон_Модуль-A.md
├── Канон_Модуль-B.md
└── ...
docs/ТЗ/
├── ТЗ_Модуль-A.md
├── ТЗ_Модуль-B.md
└── ...
docs/STACK.md
```

### Стадия 2 — Canon Review (`canon-review`)

Аудит канонов на полноту: ищем дыры в архитектуре, упущения в lifecycle, неучтённые edge cases.

**Артефакт:** `docs/CANON-REVIEW-FINDINGS.md` — список пробелов и рекомендаций. После применения — каноны обновляются и помечаются как **enterprise-ready**.

### Стадия 3 — Design Canon (`design-canon`)

Создание дизайн-системы:
- Палитра (один brand-цвет + neutrals + semantic)
- Типографика (size-driven hierarchy)
- Spacing/radii/shadows токены
- Иконки (выбор системы и правил применения)
- Motion (тон движения)
- **HTML-визуализация** для предварительной проверки

**Артефакт:**
```
docs/DESIGN-SYSTEM/
├── README.md                ← философия и правила
├── tokens.css               ← все CSS-переменные
├── BRAND-VOICE.md           ← правила копирайта
├── preview/                 ← HTML-страницы с визуализацией
│   ├── index.html           ← мастер-индекс всех страниц
│   ├── colors.html
│   ├── typography.html
│   ├── components.html
│   └── ...
└── assets/                  ← логотип, иллюстрации
```

После проверки HTML — пользователь утверждает или просит правки. Итерация быстрая: HTML, не код приложения.

### Стадия 4 — Roadmap & First Phase Prompt (`canon-roadmap`)

Декомпозиция канонов на фазы:
- Граф зависимостей между модулями
- Критический путь
- ROADMAP.md с фазами, sub-фазами, ожидаемыми артефактами
- Snapshot текущего состояния (`CURRENT-STATE.md`)
- Промпт первой фазы (самодостаточный)

**Артефакт:**
```
docs/ROADMAP.md
docs/CURRENT-STATE.md
docs/promtps/PHASE-1-{name}.md
```

### Стадия 5 — Phase Implementation (`canon-execute`)

**Здесь меняется роль и сессия.** Промпт фазы передаётся **новому агенту в чистой сессии**. Этот агент:

1. Читает промпт самостоятельно
2. Делает pre-flight: branch hygiene, проверка состояния
3. Реализует фазу с дисциплиной (transactions, idempotency, brand-compliance)
4. Открывает PR с правильным описанием

**Артефакт:** Pull Request с реализацией.

### Стадия 6 — Phase Verify (`canon-verify`)

После merge фазы (или перед merge для thorough review) — **новый агент в чистой сессии** проводит audit:

1. Читает каноны и реализацию
2. Проходит по матрице измерений
3. Находит расхождения / баги / упущения
4. Возвращает structured report

**Артефакт:** `docs/AUDIT-PHASE-{N}.md`.

### Стадия 7 — Iterate

Координатор обрабатывает audit-findings:
- 🔴 Critical → hotfix PR
- 🟡 Major → задача в backlog текущей фазы или следующей
- 🟢 Minor → ROADMAP §10.4 как acknowledged debt
- ✅ Acknowledged debt confirmed → ничего не делаем

Затем `canon-roadmap` повторно → промпт следующей фазы → 5-6 повтор.

## Сессионная модель

Это критически важно понять. **Pipeline не делается в одной сессии.** Минимум три типа сессий:

### Сессия А — Архитектурная (long-lived)
Долгоживущая сессия, где работает **архитектор**. Используются: `braindump-to-canon`, `canon-review`, `design-canon`. Задача — собрать каноны и дизайн-систему.

### Сессия Б — Координационная (long-lived)
Долгоживущая сессия, где работает **оркестратор**. Используется: `canon-roadmap`. Задача — декомпозировать каноны на фазы, писать промпты, делать code-review, обновлять ROADMAP.

### Сессия В — Реализационная (per-phase, fresh)
Короткая сессия на одну фазу. Используется: `canon-execute`. Задача — выполнить промпт. Закрывается после открытия PR.

### Сессия Г — Аудиторская (per-phase, fresh)
Короткая сессия на один audit. Используется: `canon-verify`. Задача — пройти audit-матрицу. Закрывается после возврата отчёта.

**Зачем разделение:**
- Сессия А и Б — архитектура и оркестрация. Нуждаются в **долгой памяти** проекта.
- Сессия В и Г — конкретная задача. Нуждаются в **чистом контексте** без багажа.

Если попытаться всё в одной сессии — на 4-й фазе оркестратор не сможет писать новые промпты (контекст полон), реализатор будет защищать свои предыдущие решения вместо чистого ре-факторинга.

## Анти-паттерны

### «Сначала закодим, потом задокументируем»
Канон становится копией кода. Теряется независимая проверка. Реальная архитектура остаётся в голове разработчика и исчезает с ним.

### «Один агент делает всё»
Контекст переполняется к 3-й фазе. Решения становятся ситуативными. Нет независимого audit'а.

### «Промпт — это пара абзацев»
Агент догадывается, открывает плохой PR, цикл удлиняется на день. Самодостаточный промпт — большой, но эффективный.

### «Audit делает тот же агент что и реализовал»
Slepa-точка. Найдёт только то что искал.

### «ROADMAP — это файл который мы один раз создали и забыли»
ROADMAP — живой документ, обновляется после каждой фазы. Live-track + acknowledged debt + журнал решений — три обязательных секции.

## Метрики здоровья процесса

Если pipeline работает правильно — наблюдается:

- **% PR с первого ревью без правок** растёт (промпты становятся самодостаточнее)
- **Acknowledged debt** регулярно закрывается (debt не накапливается, а перекатывается между фазами)
- **Каноны обновляются** по результатам audit'ов (живой документ, а не археология)
- **Между фазами проходит мало времени** (нет потери контекста — план в ROADMAP, не в голове)
- **Нет hotfix'ов в production за последний месяц** (pre-merge gates ловят критическое)

Если эти метрики проседают — обычно потому что один из принципов нарушен. Самые частые: «промпт сократили», «audit не делали», «канон не обновили после отклонения».
